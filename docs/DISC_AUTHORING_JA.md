# tsMuxeR のデュアルレイヤーおよび BD-R XL オーサリング追加機能

このフォークは、jaminmc/tsMuxer にマルチレイヤー（BD-R/RE DL および BD-R XL）オーサリング機能を追加します。いずれもオプトイン方式であり、デフォルトの動作は変更されません。

## 新しいオプション（mux モード）

| オプション | 用途 |
|--------|------|
| `--disc-size=<bd25\|bd50\|bd100\|bd128 \| バイト数>` | ディスク容量ガード。推定イメージがターゲットディスクに収まらない場合、mux を開始する前に中止します。 |
| `--allow-oversize` | `--disc-size` と併用し、容量超過を致命的エラーではなく警告として扱います。 |
| `--layer-break-guard=<MB>` | 各レイヤーブレイクの直後（次レイヤーの先頭。ディスクの欠陥が発生しやすい位置）に `<MB>` 分をゼロ埋めし、さらにその手前に、直後の値に比例したマージンを設けます。マージンは直後の値の 16 分の 1（当初設計の 64:4 の比率）で、最低 4 MB です。つまり `64` では従来どおり手前 4 MB、`288` では手前 18 MB になります。これらのセクタにはファイルデータが配置されません。ブレイクをまたぐファイル（通常は本編）は論理的に連続した 1 つの UDF ファイル（ギャップ前後の 2 つのエクステント）のままとなり、先読みによってシームレスに再生されます。この領域を意図的に非対称にしているのは、実機テストで次レイヤー側の欠陥が約 35 MB に達する一方、前レイヤーの末尾は正常であることが分かったためで、確保量を欠陥のある側に割り当てています。`288` を使用してください（GUI の既定値。報告されている 35 から 258 MB の欠陥領域と、欠陥管理ありディスクでずれるブレイク位置をカバーします）。`0` はフィラーなしで整列します。指定しない場合は無効です。 |
| `--layer-break-guard-before=<MB>` | 任意。ブレイクの手前側のゾーンを、比例配分による既定のマージンではなく個別に指定します。既定が非対称なのは、計測された欠陥が次レイヤーの先頭にあるためですが、一部のメディアはブレイクの手前側も弱いことがあるため、これで両側を確保できます。指定しなければ非対称の既定のままです。 |
| `--layer-break-lbn=<セクタ[,セクタ...]>` | レイヤーブレイクのセクタを設定します。BD-R/RE DL は 1 つ、100 GB BD-R XL は 2 つ、128 GB BD-R XL は 3 つ指定します（下記参照）。 |
| `--disc-capacity=<セクタ数>` | 任意。`--bdmv-to-iso` 専用。ディスクの総「Free Sectors」です。これを指定すると、レイヤーフィット配置が、ファイルを丸ごとブレイクの先へ移動してもディスクに収まるかを確認できます。指定しない場合は、ブレイク間隔から控えめな容量が導出されます。GUI はこの値を自動的に渡します。 |
| `--original-order` | 任意。`--bdmv-to-iso` 専用。ファイルをサイズの大きい順ではなく番号（再生）順に書き込みます。多数のセグメントを再生順に近い並びで保持すべきシームレスブランチングのディスク向けです。 |
| `--no-layer-fit` | 任意。`--bdmv-to-iso` 専用。レイヤーフィット配置（下記参照）を無効にし、ガード領域をまたぐファイルを常に分割します。 |

## 新しいモード: `--bdmv-to-iso`

```
tsMuxeR --bdmv-to-iso [--layer-break-guard=<MB>] [--layer-break-lbn=<セクタ[,セクタ...]>] <BDMV フォルダ> <out.iso>
```

保護のかかっていない既存の BDMV フォルダを、再 mux も再採番も行わずに、バイト単位でそのまま UDF 2.50 BD-ROM ISO にラップします。すべての `.bdjo`、`.jar`、`.clpi`、`.mpls`、`index.bdmv`、`MovieObject.bdmv`、および `CERTIFICATE/` 一式はそのままコピーされるため、BD-J メニューとすべてのクリップおよびプレイリスト参照は有効なまま保たれます（ガードバンドが移動させるのは UDF エクステントだけで、これはファイルシステムがプレーヤーから隠蔽します）。最大の `.m2ts` が最初に書き込まれ、本編タイトルがレイヤーブレイクをまたいでガードバンドを受け取ります（あるいは上記オプションの `--original-order` を使用します）。MakeMKV の補助フォルダはスキップされます。

レイヤーフィット配置（デフォルトで有効）: ガード領域をまたぐことになるファイルが、その領域の終端から次の領域まで（かつディスク容量内）に丸ごと収まる場合、そのファイルは分割されず、ブレイクの向こう側へ丸ごと配置されます。典型的な利点は 2 つあります。大きなタイトルが 2 つあるディスク（劇場版とディレクターズカット）では 1 レイヤーに 1 タイトルずつ収まり、ブレイクがちょうどその間に入ります。また、シームレスブランチングのディスクでは多数のセグメントが並べ替えられ、ブレイクがセグメントとセグメントの間に落ちます。これは市販のオーサリング済みディスクがブレイクを配置するのと同じ方法です。1 レイヤーより大きいファイルは、従来どおりガードを内包した形でブレイクをまたぎます。

ビルド後、ログと `<out.iso>.layerbreak.txt` サイドカーファイルに各ガードの位置が報告されます。内容は、ゼロ領域のセクタ範囲、対象のファイル、そのファイル内でのバイトオフセット、そしてストリームファイルの場合はブレイクのおおよその再生時刻です。GUI の節で説明するシームレスブランチングの制限に注意してください。セグメントファイル内では、時刻はそのセグメント基準になります。

再生ターゲット: 自作の BD-J ディスクを確実に再生できる環境は、ソフトウェアプレーヤー（VLC、libbluray、Kodi、PowerDVD）です。一部の据置型プレーヤーは、記録型メディア上の BD-J を制限します。

## 既存の ISO から作業する

tsMuxeR が読み取るのは BDMV フォルダであって ISO イメージではなく、ISO を直接書き換えることもありません。ただし、ISO を事前に展開する必要はありません。仮想ドライブに ISO をマウントしてドライブレターとして見えるようにし、`--bdmv-to-iso` をそのドライブに向ければ、tsMuxeR はマウントされたイメージからディスク構造を直接読み取ります。

- Windows 8、10、11: `.iso` をダブルクリックしてマウントします（例: `E:`）。
- Windows 7: 無料の仮想ドライブツール（WinCDEmu または Virtual CloneDrive）をインストールしてからマウントします。

```
tsMuxeR --bdmv-to-iso --layer-break-guard=64 --layer-break-lbn=<セクタ[,セクタ...]> E:\ out.iso
```

tsMuxeR はマウントされたドライブからファイルを読み取り、1 回のパスでガード領域付きの新しい ISO をバイト単位でそのまま、再 mux なしで書き出します。これは**暗号化されていない**素材でのみ機能します。AACS が残っている市販ディスクのイメージをマウントしても、通常のファイルとしては読み取れません。この手順が必要になるのは、デュアルレイヤーまたは BD-R XL メディアに書き込み、レイヤーブレイクのガードを適用したい場合だけです。すでに再生できるシングルレイヤーのイメージは、そのまま書き込めます。

## シームレスブランチング: その正体と、影響が及ぶもの・及ばないもの

ディスク上では、シームレスブランチングの映画は 1 つのファイルではなく、多数のセグメントファイル（`00001.m2ts`、`00002.m2ts`、...）で構成されています。プレイリストは「セグメント 3 を再生し、次に 7、次に 12」と指示する小さなスクリプトで、劇場版はあるパスを、ディレクターズカットは別のパスをたどり、大部分のセグメントを共有します。「シームレス」な部分は、再生時にプレーヤーのバッファ内で実現されます。これは再生の振り付けであって、バイト列として特別な形で保存されているものではありません。

**ブランチングが影響するもの: 配置であって、正しさではありません。** ビルドは通常、ファイルをサイズの大きい順に書き込むため、ブランチングのディスクではセグメントが再生順ではなくサイズ順に並び、低速なドライブでは多少のシーク距離が余分にかかります。まさにそのための機能が「Keep original file order」オプションです。レイヤーフィット配置では、どちらの並びでもブレイクは 2 つのセグメントの間に落ちます。これは市販のオーサリング済みディスクがブレイクを置くのと同じ位置です。また、レイヤーブレイクレポートに記載される再生時刻は、レポートが挙げるセグメントファイル基準であり、本編全体基準ではありません（上記のレポートに関する注記を参照してください）。

**ブランチングが影響できないもの: ベリファイです。** ガードチェックは、ブレイク位置の生の 2048 バイトセクタを読み取り、「このバイト列はゼロか」という 1 つの問いだけを確認します。周囲のデータが 1 つの大きな本編ファイルに属していようと、ブランチングディスクのセグメント 47 に属していようと、2 つのセグメントの間のギャップであろうと、ディスク上のある位置にゼロ帯があることは物理的な事実であり、その上に載っている再生ロジックとは無関係です。コンテンツチェックは、双方のすべてのファイルをハッシュ化して比較します。ブランチングのディスクは単にリストのファイル数が多いだけで、再生時にそれらをつなぎ合わせるプレイリストの振り付けは、どのファイルの中身も 1 バイトたりとも変えません。すべてのセグメントのハッシュが一致すれば、どのカットがどの順序で再生しようとも、内容は無事に保たれています。

たとえるなら、本を箱詰めして発送する場面を考えてください。ガードチェックは、周りにどの本があろうと、保護用の詰め物が箱の中の正しい位置にあることを確認します。コンテンツチェックは、すべての本を 1 ページずつ元の発送リストと照合します。シームレスブランチングとは、あとで読者がゲームブックのように本から本へと飛び移りながら読むという事実にすぎません。それは読み方の話であって、箱の中身や詰め物の位置を変えることはできません。

## 重要: `--layer-break-lbn` に入れる数値はどれか

`--layer-break-lbn` は、ドライブが物理的にあるレイヤーから次のレイヤーへ切り替わるセクタを、2048 バイトの LBA セクタ単位で指定するものです。記録型 BD はユーザーデータ容量を各レイヤーへ均等に分割するため、ブレイクのセクタはディスク総セクタ数の単純な分数になります。

```
BD-R/RE DL  (2 レイヤー): 総数 / 2
BD-R XL 100 (3 レイヤー): 総数 / 3  および  総数 * 2 / 3
BD-R XL 128 (4 レイヤー): 総数 / 4,  総数 * 2 / 4,  総数 * 3 / 4
```

標準的な 50 GB の BD DL では `24,438,784 / 2 = 12,219,392`（1 レイヤーあたり 25 GB）です。総数が `47,305,728` の 100 GB BD-R XL では、2 つのブレイクは `15,768,576` と `31,537,152` になります。

### 落とし穴（必読）

総数は、汎用 API が返す値ではなく、ディスクのフォーマット済み総容量から取得してください。実機の Verbatim BD-R DL で確認された例:

| 取得元 | 総セクタ数 | / 2 | 正しいか |
|--------|-----------|-----|---------|
| ImgBurn の「Disc Information, Free Sectors」 | **24,438,784** | **12,219,392** | はい、本物のブレイク位置 |
| Windows IMAPI の `TotalSectorsOnMedia` | 23,652,352 | 11,826,176 | BD-R DL では不可: 23,652,352 は欠陥管理ありの容量（BD-RE DL）で 0.8 GB 手前 |

誤った（小さい）数値を与えると、ガード領域が実際のレイヤー遷移より約 0.8 GB 手前に配置され、映像本編が欠陥の起きやすいセクタに乗ってしまいます。これはガードが防ごうとしているまさにその事態です。ディスクのフォーマット済み総容量（ImgBurn の「Free Sectors」、または `READ FORMAT CAPACITIES` の formatted もしくは maximum ディスクリプタ）を使い、レイヤー数で割ってください。

オーケストレーター側がこの数値をドライブから読み取り、`--layer-break-lbn` に渡す想定です。tsMuxeR 自体がライターに触れることは一切ありません。GUI には、ImgBurn の「Free Sectors」値をブレイクセクタへ変換する小さな計算機が用意されています。

## 推奨フロー（メニューとガード付きのマルチレイヤーディスク）

1. ターゲットディスクのフォーマット済み総容量を読み取ります（例: ImgBurn の「Free Sectors」）。レイヤー数で割ってブレイクセクタを求めます。標準的な 50 GB ディスクでは `12,219,392` です。
2. `tsMuxeR --bdmv-to-iso --layer-break-guard=64 --layer-break-lbn=<ブレイク> <BDMV フォルダ> out.iso`
3. `out.iso` をベリファイ付きで書き込みます。レイヤーブレイクはメディア側で固定されているため、ライターは同じセクタでレイヤーを切り替え、ガード領域がちょうどその位置に乗ります。

## GUI での操作（「BDMV folder -> ISO」タブ）

GUI では、コマンドラインを使わずに同じ機能を利用できます。「BDMV folder -> ISO」タブでの操作は次のとおりです。

- **BDMV folder and output ISO.** フォルダ欄には、`BDMV/`（および `CERTIFICATE/`）を含むディスクのルートを指定します。マウントした ISO ドライブもフォルダとして指定できます。
- **ディスクタイプを選ぶと Free Sectors が自動入力されます。** 一覧からディスクを選ぶと（BD-R DL 50 GB、BD-RE DL 50 GB、BD-R XL 100 GB、BD-R XL 128 GB）、その標準的な「Free Sectors」が自動で入力され、**ロック**されます。ImgBurn を実行する必要がなく、値を誤って変更することもありません。ブレイクセクタは自動で計算されます。BD-R DL と BD-RE DL が別々の項目になっているのは容量が異なるためです。空の BD-R DL は 24,438,784 セクタ（全容量）ですが、BD-RE DL は 23,652,352（欠陥管理用の予備領域を確保）です。ディスクが標準的でない場合（再フォーマットした BD-RE や、欠陥管理なしで焼いた BDXL など）は、**Enter Free Sectors manually (advanced)** にチェックを入れて欄のロックを解除し、ImgBurn がお使いの実際のディスクについて表示する値を入力してください。
- **Layer-break guard (after break).** ガードサイズを MB 単位で指定します。入力値に応じて色分けのヒントが表示されます。既定値は 288 MB です。報告されている実際の欠陥領域は 35、64、258 MB 付近に集中しており、さらに欠陥管理ありのディスクでは実際のレイヤー切り替えが計算上のブレイクの最大 128 MB 後ろに位置することがあるため、288 はこれらすべてをカバーします。欠陥領域が 1 GB を超えるまれなメディアにはより大きな値が必要です（入力欄は 9999 まで受け付けます）。**Also fill before the break (advanced)** にチェックを入れると 2 つ目の入力欄が表示され、ブレイクの両側が弱いメディア向けに、その手前側のゾーンも確保できます。
- **Keep original file order (seamless branching).** 通常、ファイルはサイズの大きい順に書き込まれ、本編がレイヤーブレイク上に乗ってガードを受け取ります。シームレスブランチングを使うディスクは、本編を順番に再生される多数のセグメントファイルとして保存しています。ここにチェックを入れるとファイルが番号（再生）順のまま保たれ、元のディスクと同様に、セグメントが再生される順序に近い並びで物理的に配置されます。どちらのモードでも、ブレイクをまたぐことになるファイルが次のレイヤーに丸ごと収まる場合はそこへ丸ごと配置されるため、ブレイクはファイルを分割せず、ちょうどファイルとファイルの間に落ちます。これは市販のオーサリング済みディスクがレイヤーブレイクを配置するのと同じ方法です。
- **Layer break report.** ビルド後、各ガードがどこに配置されたかがログに表示されます。どのストリームファイルか、そのファイル内でのバイトオフセット、そして（通常の単一ファイルの本編であれば）再生時刻が示されるため、プレーヤーでどこを重点的に確認すればよいかが分かります。同じ情報は、イメージの隣に `<name>.iso.layerbreak.txt` として書き出されます。現在の制限: シームレスブランチングのディスクでは、表示される時刻はそのセグメントファイル内での位置であり、本編全体での位置ではありません。本編全体での時刻を出すにはディスクのプレイリストを信頼する必要がありますが、多くのディスクはダミーのプレイリストを収録しているため、ツールはセグメントを報告し、チャプターで位置を特定できるようにしています。
- **Fit estimate.** フォルダ、ディスクタイプ、「Free Sectors」、ガードを入力していくと、推定イメージサイズがディスク容量と照らし合わせてリアルタイムに表示されます。収まる場合は緑（残り容量も表示）、収まらない場合は赤（超過分も表示）で示されます。ガードバンドも推定に含まれるため、バッファを増やすと即座に反映されます。
- **Build ISO** を押すと、入力した値で同じ `--bdmv-to-iso` コマンドが実行されます。

## GUI が自動入力する標準容量

ディスクタイプの一覧は、空のディスクの「Free Sectors」値（2048 バイトセクタ）を自動入力します。これは ImgBurn が通常の空ディスクについて表示するのと同じ数値で、値を誤って変更しないよう欄はロックされます。

| ディスク | Free Sectors | 備考 |
|------|-------------|------|
| BD-R DL 50 GB | 24,438,784 | 追記型、全容量 |
| BD-RE DL 50 GB | 23,652,352 | 書き換え型、欠陥管理用の予備領域を確保 |
| BD-R XL 100 GB | 47,305,728 | BDXL、欠陥管理あり |
| BD-R XL 128 GB | 60,403,712 | BDXL、欠陥管理あり |

これらは（通常の）欠陥管理ありの容量です。数値はメディアに関する事実で、実際の Verbatim ディスクから読み取り（ImgBurn の「Free Sectors」）、Blu-ray/BDXL の仕様に一致します。欠陥管理のフォーマットをせずに焼いたディスクはより大きな全容量になります（たとえば 100 GB なら 48,878,592、128 GB なら 62,500,864）。ImgBurn がお使いのディスクについてそれらを表示する場合は、**Enter Free Sectors manually (advanced)** にチェックを入れて入力してください。特に BD-RE の容量はディスクのフォーマット方法によって変わるため、欄のロックを解除できるようになっています。

## 欠陥管理ありディスク（実機での計測）

欠陥管理（スペアエリア）ありでフォーマットした BD-R DL は、通常のディスクとは挙動が異なります。以下の 2 つの効果はどちらも、データシートからではなく、実際に焼いたディスクで計測されたものです。

- **容量。** スペアエリアは、そのサイズ分だけ正確に Free Sectors を減らします。ImgBurn の Disc Definition Structure には 32 セクタ単位のクラスタで記載されており、ImgBurn の Format Capacities にある `TDP` 値がスペアの合計です（たとえば `TDP: 24576` クラスタ = 786,432 セクタ = 24,438,784 と 23,652,352 のちょうどの差分）。したがって、欠陥管理ありの BD-R DL には BD-RE DL の容量項目を使用します。
- **レイヤー切り替えは「容量をレイヤー数で割った位置」ではありません。** スペアエリアはレイヤー間で不均等に割り当てられることがあります（実際の ImgBurn でのフォーマットでは、レイヤー 0 が ISA0 4,096 + OSA0 6,144、レイヤー 1 が ISA1 8,192 + OSA1 6,144 になりました）。その結果、各レイヤーが保持するユーザーデータ量は異なり、実際の切り替え位置は「レイヤーサイズから `(ISA0+OSA0) x 32` セクタを引いた位置」になります。実機のディスクでは、これが容量/2 の 128 MB **後ろ**で計測されました（読み取りタイミングのスキャンで、遷移点におけるドライブの再フォーカスによる一時停止が確認できます）。BDXL 100/128 GB のフォーマットはレイヤーごとのスペアが対称なので、こちらは均等割りが正確に成り立ちます。
- **対処方法。** 特別な対応は不要です。既定の 288 MB ガードは、ずれた切り替え位置を余裕をもってカバーしており、これはエンドツーエンドで検証済みです（ディスクは遷移点をまたいでシームレスに再生されます）。欠陥管理ありのディスクで小さなカスタムガードを使う場合に限り、上記の式を使って `--layer-break-lbn` でブレイクを手動配置してください。
- **映像用途における欠陥管理のトレードオフ。** ドライブは書き込みながらベリファイを行い、不良クラスタを自動的にスペアエリアへ再マップします。このため書き込み速度はおよそ半分になります。再マップされたクラスタは物理的に別の場所へ移されるため、再生中にそこを読むとヘッドシークが発生します。データディスクでは問題になりませんが、映像では、ガードバンド付きの通常（欠陥管理なし）の書き込みのほうが滑らかな結果になります。欠陥管理とガードバンドは別々の問題を解決するものです。再マップはデータを保全し、ガードはシークを起こさず映像を途切れさせません。

## 焼き込んだディスク上のガードサイズ（セクタ境界への整列）

ガードは常に 2048 バイトのセクタ単位で埋め、正確なバイト位置ではなく、本編の最後のデータエクステントの末尾から始まります。そのためレイヤーブレイク周辺のゼロ領域は、セクタ境界とエクステント境界に合わせられます。完成したディスク上では、設定した値にごく近くはなりますが（およそ 1 MB 以内）、バイト単位で厳密に一致するわけではなく、同じかやや大きめになる傾向があり、意味のあるほど小さくなることはありません。たとえば片側 160 MB と設定しても、読み戻すと約 160.2 MB になることがあります。これは正常な境界合わせであり、不具合ではありません。ガードはレイヤー遷移部の約 35 MB の欠陥領域を覆うために数十から数百 MB の規模で確保されるため、1 MB 未満のずれは再生にまったく影響しません。したがって、完成したディスクのゼロ領域が入力した値と正確に一致しなくても、それは想定どおりの動作です。

## BD-R XL（100 / 128 GB）のプレーヤー互換性

多くの Blu-ray プレーヤーは 100 GB や 128 GB の BD-R XL ディスクをそもそも読み取れず、特定のプレーヤーで再生できる保証はありません。イメージを 66 GB 程度（最初の 2 レイヤー）に収めてディスクをファイナライズすると、一部のプレーヤーで再生できる可能性は高まりますが、66 GB でも再生は保証されません。フル 100 GB または 128 GB には、大容量記録型メディアを明示的にサポートする新しめのプレーヤーが必要です。Verbatim 製メディアと低速（2 倍速）での書き込みが最も安定した結果になります。これは ISO 側ではなくプレーヤー側の制限ですので、ご自身の機器でテストしてください。
