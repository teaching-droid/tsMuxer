# tsMuxeR のデュアルレイヤーおよび BD-R XL オーサリング追加機能

このフォークは、jaminmc/tsMuxer にマルチレイヤー（BD-R/RE DL および BD-R XL）オーサリング機能を追加します。いずれもオプトイン方式であり、デフォルトの動作は変更されません。

## 新しいオプション（mux モード）

| オプション | 用途 |
|--------|------|
| `--disc-size=<bd25\|bd50\|bd100\|bd128 \| バイト数>` | ディスク容量ガード。推定イメージがターゲットディスクに収まらない場合、mux を開始する前に中止します。 |
| `--allow-oversize` | `--disc-size` と併用し、容量超過を致命的エラーではなく警告として扱います。 |
| `--layer-break-guard=<MB>` | 各レイヤーブレイクの直後（次レイヤーの先頭。ディスクの欠陥が発生しやすい位置）に `<MB>` 分をゼロ埋めし、さらにその手前に固定の 4 MB のマージンを設けます。これらのセクタにはファイルデータが配置されません。ブレイクをまたぐファイル（通常は本編）は論理的に連続した 1 つの UDF ファイル（ギャップ前後の 2 つのエクステント）のままとなり、先読みによってシームレスに再生されます。この領域を意図的に非対称にしているのは、実機テストで次レイヤー側の欠陥が約 35 MB に達する一方、前レイヤーの末尾は正常であることが分かったためで、確保量を欠陥のある側に割り当てています。`64` を使用してください（マージン込みで約 35 MB をカバーします）。`0` はフィラーなしで整列します。指定しない場合は無効です。 |
| `--layer-break-lbn=<セクタ[,セクタ...]>` | レイヤーブレイクのセクタを設定します。BD-R/RE DL は 1 つ、100 GB BD-R XL は 2 つ、128 GB BD-R XL は 3 つ指定します（下記参照）。 |

## 新しいモード: `--bdmv-to-iso`

```
tsMuxeR --bdmv-to-iso [--layer-break-guard=<MB>] [--layer-break-lbn=<セクタ[,セクタ...]>] <BDMV フォルダ> <out.iso>
```

保護のかかっていない既存の BDMV フォルダを、再 mux も再採番も行わずに、バイト単位でそのまま UDF 2.50 BD-ROM ISO にラップします。すべての `.bdjo`、`.jar`、`.clpi`、`.mpls`、`index.bdmv`、`MovieObject.bdmv`、および `CERTIFICATE/` 一式はそのままコピーされるため、BD-J メニューとすべてのクリップおよびプレイリスト参照は有効なまま保たれます（ガードバンドが移動させるのは UDF エクステントだけで、これはファイルシステムがプレーヤーから隠蔽します）。最大の `.m2ts` が最初に書き込まれ、本編タイトルがレイヤーブレイクをまたいでガードバンドを受け取ります。MakeMKV の補助フォルダはスキップされます。

再生ターゲット: 自作の BD-J ディスクを確実に再生できる環境は、ソフトウェアプレーヤー（VLC、libbluray、Kodi、PowerDVD）です。一部の据置型プレーヤーは、記録型メディア上の BD-J を制限します。

## 既存の ISO から作業する

tsMuxeR が読み取るのは BDMV フォルダであって ISO イメージではなく、ISO を直接書き換えることもありません。ただし、ISO を事前に展開する必要はありません。仮想ドライブに ISO をマウントしてドライブレターとして見えるようにし、`--bdmv-to-iso` をそのドライブに向ければ、tsMuxeR はマウントされたイメージからディスク構造を直接読み取ります。

- Windows 8、10、11: `.iso` をダブルクリックしてマウントします（例: `E:`）。
- Windows 7: 無料の仮想ドライブツール（WinCDEmu または Virtual CloneDrive）をインストールしてからマウントします。

```
tsMuxeR --bdmv-to-iso --layer-break-guard=64 --layer-break-lbn=<セクタ[,セクタ...]> E:\ out.iso
```

tsMuxeR はマウントされたドライブからファイルを読み取り、1 回のパスでガード領域付きの新しい ISO をバイト単位でそのまま、再 mux なしで書き出します。これは**暗号化されていない**素材でのみ機能します。AACS が残っている市販ディスクのイメージをマウントしても、通常のファイルとしては読み取れません。この手順が必要になるのは、デュアルレイヤーまたは BD-R XL メディアに書き込み、レイヤーブレイクのガードを適用したい場合だけです。すでに再生できるシングルレイヤーのイメージは、そのまま書き込めます。

## 重要: `--layer-break-lbn` に入れる数値はどれか

`--layer-break-lbn` は、ドライブが物理的にあるレイヤーから次のレイヤーへ切り替わるセクタを、2048 バイトの LBA セクタ単位で指定するものです。記録型 BD はユーザーデータ容量を各レイヤーへ均等に分割するため、ブレイクのセクタはディスク総セクタ数の単純な分数になります。

```
BD-R/RE DL  (2 レイヤー): 総数 / 2
BD-R XL 100 (3 レイヤー): 総数 / 3  および  総数 * 2 / 3
BD-R XL 128 (4 レイヤー): 総数 / 4,  総数 * 2 / 4,  総数 * 3 / 4
```

標準的な 50 GB の BD DL では `24,438,784 / 2 = 12,219,392`（1 レイヤーあたり 25 GB）です。総数が `47,305,728` の 100 GB BD-R XL では、2 つのブレイクは `15,768,576` と `31,537,152` になります。

### 落とし穴（必読）

総数は、汎用 API が返す値ではなく、ディスクのフォーマット済み総容量から取得してください。実機の Verbatim BD-R DL で確認された例:

| 取得元 | 総セクタ数 | / 2 | 正しいか |
|--------|-----------|-----|---------|
| ImgBurn の「Disc Information, Free Sectors」 | **24,438,784** | **12,219,392** | はい、本物のブレイク位置 |
| Windows IMAPI の `TotalSectorsOnMedia` | 23,652,352 | 11,826,176 | いいえ、部分的なフォーマット容量ディスクリプタで 0.8 GB 手前 |

誤った（小さい）数値を与えると、ガード領域が実際のレイヤー遷移より約 0.8 GB 手前に配置され、映像本編が欠陥の起きやすいセクタに乗ってしまいます。これはガードが防ごうとしているまさにその事態です。ディスクのフォーマット済み総容量（ImgBurn の「Free Sectors」、または `READ FORMAT CAPACITIES` の formatted もしくは maximum ディスクリプタ）を使い、レイヤー数で割ってください。

オーケストレーター側がこの数値をドライブから読み取り、`--layer-break-lbn` に渡す想定です。tsMuxeR 自体がライターに触れることは一切ありません。GUI には、ImgBurn の「Free Sectors」値をブレイクセクタへ変換する小さな計算機が用意されています。

## 推奨フロー（メニューとガード付きのマルチレイヤーディスク）

1. ターゲットディスクのフォーマット済み総容量を読み取ります（例: ImgBurn の「Free Sectors」）。レイヤー数で割ってブレイクセクタを求めます。標準的な 50 GB ディスクでは `12,219,392` です。
2. `tsMuxeR --bdmv-to-iso --layer-break-guard=64 --layer-break-lbn=<ブレイク> <BDMV フォルダ> out.iso`
3. `out.iso` をベリファイ付きで書き込みます。レイヤーブレイクはメディア側で固定されているため、ライターは同じセクタでレイヤーを切り替え、ガード領域がちょうどその位置に乗ります。

## BD-R XL（100 / 128 GB）のプレーヤー互換性

多くの Blu-ray プレーヤーは 100 GB や 128 GB の BD-R XL ディスクをそもそも読み取れず、特定のプレーヤーで再生できる保証はありません。イメージを 66 GB 程度（最初の 2 レイヤー）に収めてディスクをファイナライズすると、一部のプレーヤーで再生できる可能性は高まりますが、66 GB でも再生は保証されません。フル 100 GB または 128 GB には、大容量記録型メディアを明示的にサポートする新しめのプレーヤーが必要です。Verbatim 製メディアと低速（2 倍速）での書き込みが最も安定した結果になります。これは ISO 側ではなくプレーヤー側の制限ですので、ご自身の機器でテストしてください。
